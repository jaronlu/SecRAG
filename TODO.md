# 待办：项目还缺什么

站在"找工作有亮点、有实际使用价值"的目标排查。当前代码质量本身没问题（99 个测试全过、ruff 无告警、33 次真实提交），缺口主要在"别人打开这个仓库能不能看懂、能不能跑起来验证"这一层。

## 高优先级

1. **没有 README.md**
   仓库根目录完全没有 README，只有面向 Claude Code 的 `CLAUDE.md`（工作流约束，不是项目介绍）。任何人打开 GitHub 仓库第一眼看到的应该是：这是什么项目、解决什么问题、怎么跑起来、架构是什么样。现在这些信息全部隐藏在 `~/llm-wiki` 的设计文档里，仓库本身对外是"哑的"。这是当前最大的缺口——`jaronlu-docs` 站点上的 secrag.md 已经把这些信息写清楚了，应该把对外可讲的部分（不含内部设计文档细节）搬一份精简版进 README。

2. **检索评估脚本跑不出真实数字**
   `scripts/evaluate_retrieval.sample.json` 里 4 条评估样本的 `relevant_doc_ids` 全部是 `replace_me_xxx_chunk_id` 占位符，没有真实 chunk id，`evaluate_retrieval.py` 目前只能在占位数据上跑通逻辑，产出不了任何有意义的召回率/精确率数字。这直接对应站点上"引用准确率等指标未经生产环境实测验证"的免责声明——如果想把这条免责声明往前推一步（哪怕只是"在自建小型测试集上跑出的参考数字，非生产环境"），需要先把这几条样本换成 ingestion 后真实产出的 chunk id，再跑一次拿到实测结果。这是把"设计目标"变成"至少有一次真实测量"的最低成本路径。

3. **没有任何方式能让别人在 5 分钟内验证你说的东西**
   `src/api/main.py` 有 `/v1/qa` 和 `/v1/assistant/qa` 两个端点，但没有一个能一键跑起来的 demo 脚本或 README 里的 curl 示例。别人（面试官、招聘方技术负责人）如果想验证"这个 Agent 真的能跑六节点工作流"，现在得自己读代码拼请求体。补一个 `scripts/demo.py`（或 README 里的 3~5 行 curl 示例 + 示例返回）能把"我说它能跑"变成"你自己试试看"。

## 中低优先级

4. **仓库当前是 Private，站点没有链接**
   `gh repo view` 确认仓库可见性是 PRIVATE。这是此前规划里明确标注"暂不处理，留给你决定"的项——如果决定公开并在站点加链接，上面 3 条都应该在公开前处理完，否则公开后第一印象反而是负面的（空 README + 占位评估数据）。

5. **没有 LICENSE 文件**
   个人项目公开与否都建议加一个（如 MIT），否则默认"保留所有权利"，对愿意看代码的招聘方不友好。

6. **没有 CI**
   `.github/` 目录不存在，99 个测试目前只在本地手动跑。哪怕只是一个跑 `pytest` + `ruff check` 的最简 GitHub Actions workflow，也能把"36+ 单测"这个卡片式描述变成"每次 push 都有徽章证明测试是绿的"。

7. **没有 `.env.example`**
   `src/config.py` 里所有配置项都有默认值，代码本身不依赖 `.env.example` 就能跑，但对第一次看这个项目的人来说，缺一份"这些环境变量是什么、需要填什么"的示例文件，会增加理解成本。

8. **`docs/design` 是指向仓库外部路径的软链接**
   指向 `/Users/ryan/llm-wiki/raw/10-AI/projects/agentic-rag-securities`，只在你本机有效。如果仓库要公开，这个软链接对别人是断的——需要决定是把设计文档一并搬进仓库，还是在 README 里说明"设计文档未披露，以下是精简版架构说明"。
